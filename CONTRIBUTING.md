# Contributing

## Branching

Use short-lived branches from `main` for focused work. Keep changes small enough to review quickly and easy to revert if scope drifts.

## Commit Messages

Use concise, imperative commit messages such as `Add mobile scaffold` or `Document MVP constraints`.

## Using Codex Safely

- Start by reading `docs/business-criteria.md` and `docs/founder-contract.md`.
- Prefer the smallest implementation that proves the workflow.
- Do not add integrations, auth, automation, or assistant-style AI behavior unless explicitly requested.
- Flag scope changes clearly before landing them.

## Pull Requests

PR descriptions should state:

- what changed
- why it changed
- any user-visible behavior
- any scope impact relative to the MVP
- any follow-up work intentionally deferred
